# /frontciti shape

## Purpose

Plan UX and visual direction before implementation. Produces a design brief, not code.

## Use When

Use when scope, audience, content, register, visual lane, states, or success criteria are unclear enough that coding would be guesswork.

## Inputs To Inspect

`PRODUCT.md`, `DESIGN.md`, existing routes/components/styles, screenshots, user brief, target audience, content/data shape, platform, and constraints.

## Workflow

1. Identify register, platform, audience, primary user action, realistic data ranges, states, and constraints.
2. Ask only the questions that cannot be inferred safely. Prefer one focused round.
3. Define visual lane, color strategy, density, motion level, references, and anti-references.
4. For visually important net-new work, use image-first probes when image generation is available.
5. Produce a compact or structured design brief.
6. Stop for confirmation before implementation.

## Output Expectations

Include feature summary, primary user action, design direction, scope, layout strategy, states, interaction model, content/media requirements, recommended references, and open questions.

## Anti-Patterns

- Writing code before direction is confirmed.
- Asking a long questionnaire when the repo already answers it.
- Treating a generated visual as a replacement for accessibility, copy, states, or real UX.

## Acceptance Checklist

- Brief is specific enough to build from.
- Ambiguities are resolved or clearly marked.
- Register and visual lane are explicit.
- Implementation has not started without confirmation.
